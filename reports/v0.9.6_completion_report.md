## 1. 设计与执行对齐

本版本严格按 `reports/v0.9.6_design.md` 的 DAG 执行：

```
T1 baseline harness ─┐
                     ├─→ T4 N+1 batch ─┐
T2 core tests ───────┤                 ├─→ T8 regression + report
                     ├─→ T3 conn reuse ┤
                     └─→ T5 remove _RUNS┤
T6 rate limiter  ────(independent)─────┤
T7 warnings cfg  ────(independent)─────┘
```

根节点 T1（基线）与 T2（测试网）先行，所有高风险任务（T3/T4/T5）都在其之后消费"基线"与"测试网"。这落实了设计中的 C1/C2 前置约束：
- **C1** — 没有基线就无法量化性能收益（先测量再优化）。
- **C2** — 没有回归网就不能安全做行为变更（先测试再改）。

## 2. 逐任务完成情况

### T1 — 性能基线采集工具 ✅

- 新增 `backend/app/tests/benchmark.py`。
- 对 5 条控制回路路径采样 p50/p95/p99/mean，输出 JSON。

- 产物：`reports/perf_baseline_before.json`、`reports/perf_baseline_after.json`。

### T2 — 核心模块单元测试 ✅

新增 4 个测试文件 + 共享夹具 `conftest.py`（隔离临时 DB）：

| 模块 | 语句数 | 覆盖率 |
|------|--------|--------|
| `capsule_store.py` | 117 | **100%** |
| `command_loop.py` | 31 | **100%** |
| `policy_gate.py` | 33 | **100%** |
| `retrieval.py` | 67 | **94%** |
| **核心 4 合计** | **248** | **98%** |

- 目标是核心模块 ≥ 70% 行覆盖，实际 98%，远超目标。
- `retrieval.py` 未覆盖的 4 行是 FTS 异常兜底分支（`except: return []`）和空查询早退，属于难以稳定触发的防御性分支，不影响主路径证明。

**对设计文档"45% 覆盖率"表述的修正**：设计初稿曾写"whole-backend 45%"，经算术核验不成立——全后端 5156 行，核心 4 模块仅 248 行，即使 100% 也只带动全局约 +5pt。真正对 B2/C 去风险有意义的是**被改动模块**的覆盖率，而非全局。因此本版本目标口径已更正为"核心模块覆盖率"，实测 98%。全局覆盖率提升是副产品，不作为承诺指标。

### T3 — 线程本地连接复用 + WAL ✅

- `backend/app/db.py`：`get_conn()` 改为按 `(线程, 解析后路径)` 缓存连接；新增 `close_all()` 供测试 teardown 释放句柄。
- 启用 `PRAGMA journal_mode=WAL` + `synchronous=NORMAL` + `busy_timeout=5000`。
- 边界处理：
  - FastAPI 在线程池中运行同步端点，共享单连接会触发 `check_same_thread` 错误 —— 因此按线程缓存，绝不跨线程复用。
  - 测试在用例间切换 `WANWEI_MEMORY_DB` 并 unlink 临时文件 —— 因此按解析路径缓存，路径变化透明打开新句柄，并在 teardown `close_all()`。

**连接复用收益的诚实修正**：设计文档曾判断"本地 SQLite 的 `connect()` 亚毫秒，连接复用只是小幅收益"。实测数据推翻了这一保守判断——见 §3。原因是旧 N+1 路径**每次 `get_capsule` 都新开一个连接**（一次检索约 25 次 connect），因此 T3 与 T4 在检索路径上是**叠加放大**的，收益比单看语句数更大。这是好的意外，但仍如实标注：write/workflow 等非 N+1 路径的连接复用收益相对温和。

发现并修复的 T3 副作用：
- `workflow/persistence.py` 原有 7 处 `conn.close()`，在"每次新连接"模型下无害，但缓存后会关掉共享句柄导致 `Cannot operate on a closed database`。已移除全部 7 处——连接生命周期现由 `db.py` 统一拥有。
- `-W always` 下 `ResourceWarning: unclosed database`：实测 T3（缓存）在全量套件下为 **16 条**，而**原始每次新建连接为 58 条**。即 T3 实际**减少**了未关闭连接告警（复用而非泄漏）。剩余 16 条来自 TestClient 工作线程内打开、主线程 `close_all()` 触及不到的连接，且仅在 `-W always` 下出现，默认运行为 0。结论：T3 对资源管理是净改善，不是回归。

### T4 — N+1 查询批量优化 ✅（用 query count 证明）

- `capsule_store.py` 新增 `get_capsules_batch()`（单条 `IN (...)` 查询）和 `bump_usage_batch()`（单条 `executemany` + 一条聚合审计）；抽出 `_row_to_capsule()` 共享反序列化。
- `retrieval.py` 的 `search_capsules()` 重写为：批量取候选 → 计算 → 批量回写 usage_count。
- 新增 `backend/app/tests/test_n1_query_count.py`，用 sqlite trace 回调**计数 SQL 语句**：

| 指标 | 旧路径（每候选） | 新路径（批量） |
|------|------------------|----------------|
| 20 候选 / top_k=5 检索的 DB 语句数 | ≈ 45 | **9** |
| 语句数是否随候选数增长 | 是（O(N)） | **否（常数）** |

- 关键证明是**计数**而非计时，因此与硬件无关、可在单测中稳定断言：
  - `test_search_query_count_is_constant`：断言检索语句数为个位数。
  - `test_search_query_count_decoupled_from_candidate_size`：候选从少到多，语句数不随之线性增长。
  - `test_usage_count_still_increments_after_batch`：批量回写后 usage_count 语义不变。

### T5 — 移除 workflow `_RUNS` 内存 fallback ✅

- v0.9.5 已将 workflow run 持久化到 SQLite，并保留 `_RUNS` 作为迁移期 fallback。
- 持久化已由 5 个测试验证稳定，本版本删除 `_RUNS` 字典及 `create_run` 的双写、`get_run` 的内存兜底。
- `create_run` / `get_run` 现在纯数据库读写；5 个持久化测试仍全绿。

### T6 — 单进程内存令牌桶限流 ✅

- 新增 `backend/app/security/rate_limit.py` + `RateLimitMiddleware`（注册为最外层，先于 auth/body 处理）。
- 每 (IP, endpoint) 令牌桶模型：容量 = 每分钟额度（突发上限），补充速率 = 额度/60（持续速率），空桶返回 429。
- 限流端点与额度（按成本/风险）：

| 端点 | 每分钟额度 | 理由 |
|------|-----------|------|
| `/model-gateway/test` | 10 | 外部 + 昂贵，最紧 |
| `/workflow/runs` | 30 | 创建持久状态，中等 |
| `/memory/v2/search` | 60 | 读/查询路径，较松 |
| `/memory/v2/capsules` | 60 | 读/查询路径，较松 |

- **LRU 上限 10000 IP**：限流器自身不会因海量不同源 IP 洪泛而成为内存耗尽（DoS）向量。
- **明确边界**：单进程内存限流。多 worker/多副本下有效额度 = 配置额度 × worker 数。**跨进程共享限流（Redis/中心存储）留到 v1.0**。
- 新增 `backend/app/tests/test_rate_limit.py`（7 项）：突发触发 429、时间补充恢复、端点隔离、IP 隔离、LRU 上限、中间件集成。

### T7 — pytest warnings 治理 ✅

- 新增 `pytest.ini`：
  - 第三方废弃告警（FastAPI/Starlette 的 `asyncio.iscoroutinefunction`）过滤为 ignore。
  - **项目代码（`backend.app`）的 DeprecationWarning 提升为 error**，防止自有代码再次引入废弃用法。
- 治理前实测：项目代码 warnings = **0**，剩余 656 条全部来自第三方库。
- 不魔改 site-packages；第三方告警只记录、不修改依赖源码。

## 3. 性能 before/after（同机可复现，非绝对承诺）

数据来源：`reports/perf_baseline_before.json`（还原 T3/T4 前代码）与 `reports/perf_baseline_after.json`（v0.9.6）。100 次采样，10 次 warmup，50 个 seed capsule，隔离临时 DB。

| 路径 | before p95 | after p95 | 变化 |
|------|-----------|----------|------|
| `capsule_search_zh` | 9.59 ms | **0.74 ms** | **≈ 13× / −92%** |
| `command_loop` | 5.89 ms | **0.78 ms** | **≈ 7.5× / −87%** |
| `capsule_write` | 0.86 ms | 0.24 ms | ≈ 3.6× |
| `workflow_create_run` | 0.72 ms | 0.26 ms | ≈ 2.7× |
| `policy_evaluate` | 0.003 ms | 0.003 ms | 持平 |

诚实解读：
- 最大收益在 `capsule_search_zh` 和 `command_loop`（后者内部调用检索）——正是 N+1 影响的路径，与 T4 目标一致。
- 该延迟降幅**大于**单看语句数（45→9，约 5×）的预期，原因是旧路径**每次 `get_capsule` 都新开连接**，T3（连接复用）与 T4（批量）在检索路径上叠加。
- `capsule_write` / `workflow_create_run` 也有温和改善，主要来自 T3 连接复用；`policy_evaluate` 是纯函数，无 DB，故持平。
- **这些是同机相对对比，受硬件/磁盘/Python 版本影响，不作为绝对性能承诺。** 权威证据仍是 T4 的 query count（与硬件无关）。
