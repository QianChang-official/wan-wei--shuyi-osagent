> 分支：`v0.9.6-rate-limit-and-test-hardening`
> 口径：v0.9.6 只做 **A-core + B1**。不做 B2（async/aiosqlite、cache），不做 C（拆 main.py、Repository/DI），不做 JWT/RBAC/SQLCipher。

## 1. 排序结论（B→A→C 的数学复核）

用户初始意向为严格 **B → A → C**。基于代码实测指标建模后，结论是把 B 拆成 **B1（安全的性能项）** 与 **B2（有风险的性能项）**，得到可行序：

```
(B1 + A-core)  →  B2  →  C
   v0.9.6          v0.9.7   v1.0
```

### 1.1 两条不可违背的结构性前置

- **C1（可证明性）**：没有基线就无法量化性能收益。基线属于 A。先优化后测量 = 无法给出可证明的收益数字。
- **C2（安全性）**：没有回归网就无法安全地做改变行为语义的重构。async 会触及全部 20 处 `get_conn` 调用点；能抓回归的测试属于 A。

因此 A 产出 B/C 消费的两样东西：**测量基线** 与 **测试安全网**。

### 1.2 期望工作量模型

对每个阶段：基础工作量 `E`，回归风险 `ρ`，实现失败率 `f = ρ·(1 − g·c)`（`g=0.8` 为测试防护效率，`c` 为被触及模块覆盖率），返工 `k`，期望工作量 `EE = E + f·k`。

| 阶段 | E(h) | ρ | k(h) |
|---|---|---|---|
| A（tests/baseline/remove `_RUNS`/pool/batch） | 15 | 0.15 | 5 |
| B（async + cache + N+1） | 19 | 0.55 | 12 |
| C（拆 61 端点 + Repository + DI） | 25 | 0.60 | 15 |

覆盖率：现状 `c₀=0.15`（被触及数据层模块），A 后 `c_A=0.45`（**被触及模块** 覆盖率，非全局）。

- **A→B→C**：A EE=15.7；B(c=0.45) EE=23.2；C(c=0.45) EE=30.8 → **≈69.6h**，B 收益完全可证明（d=1.0）。
- **B→A→C（严格）**：B(c=0.15) EE=24.8 +4h 耦合=28.8，无基线 d=0.6；A′≈18.7；C 30.8 → **≈78.2h**。
- **Δ ≈ +8.6h（约 +12%），且 B 性能收益可证明性下降约 40%（d 1.0→0.6）**。差额完全由 C1/C2 解释：B-first 把风险预算花在未测、未测量的基座上。

### 1.3 触发严格 B-first 的条件（灵敏度）

只有当"立即交付性能"的时间价值 > (+8.6h + 可证明性损失) 才成立，即 **存在正在失败的硬延迟门槛**。现状证据相反：Arena 16/16、`unsafe_autonomy_rate=0.0`、设计目标 `p95 ≤ 500ms（控制回路，除模型生成）` 已满足。**触发条件缺失**（已与用户确认：无硬延迟门槛）。

### 1.4 关键洞察：B 非单体

- **B1 = N+1 批量修复 + 连接复用/WAL**：ρ≈0.2，自证（query 计数下降），无需基线即可证明 → 归入 v0.9.6。
- **B2 = async/aiosqlite 全迁移 + cache**：ρ≈0.55，真正有风险，需要 C1+C2 → 推迟到 v0.9.7。

于是"性能优先"的直觉在 **不违背前置** 的前提下大部分被满足：把 B1 前移，仅推迟 B2。

## 5. 关键度量的诚实修正

### 5.1 覆盖率口径修正

后端全量 **5156 LOC**；core-4 模块（`capsule_store` 193 + `retrieval` 87 + `policy_gate` 105 + `command_loop` 67）合计约 **452 LOC**。即使 100% 覆盖，也只把全局覆盖率抬升约 9pt（~15%→~24%）。

因此 **"全局 45%" 不可从 core-4 单独达成**。真正对 de-risk B2/C 有意义的是 **被触及模块覆盖率**，不是全局：

- 目标：**core-4 行覆盖 ≥ 70%**（这是 B2 消费的安全网）；
- 副产物：**全局 ~24–30%**。

（§1.2 模型中的 `c_A=0.45` 指的正是数据层 **被触及模块** 覆盖率，模型成立；此前仅措辞不精确。）

### 5.2 N+1 收益（无需基线即可验证）

`retrieval.search_capsules(top_k=5)` 全命中路径下（每次成功累加后 `len(out)>=top_k` 时 break，处理约 5 行）：

- 每行：`get_capsule`(1 SELECT) + `update_capsule`。
- `update_capsule` = `get_capsule`(1) + UPDATE(1) + `record`(建表+INSERT+commit) + `get_capsule`(1)。
- 针对 capsule 表的 **SELECT** ≈ 每行 3 次 → **O(N)** 读放大。

批量路径：`_fts_rows`(1) + `_substring_rows`(CJK 时 1) + `get_capsules_batch`(1 IN SELECT) + `executemany` 批量 usage 更新 + 1 条聚合审计。**针对 capsule 表的 SELECT 降为 1，即 O(N)→O(1)。**

诚实边界：这是 **query 计数**（读放大）证明，不是墙钟计时。收益随结果数 R 增长而放大；R 小时倍数小，R 大时倍数大。验收断言 capsule-SELECT 计数 **与 R 无关**（robust 的 O(1) 证明），而非声称固定加速比。

### 5.3 连接复用 + WAL 的诚实说明

本地 SQLite 文件 `connect()` 亚毫秒级，连接池延迟收益有限，**不夸大**。可辩护的价值是：

- **WAL 模式**：更好的并发读写；
- **线程本地复用**：FastAPI 同步端点跑在线程池，朴素共享连接会触发 `check_same_thread` 错误（那才是回归）——线程本地是正确解法。

T3 定位为 **小而安全的正确性/并发改进**，不作为头条性能数字。若 T1 基线显示是噪声，报告中如实写"收益不明显"，不编造加速比。

### 5.4 限流器边界

per-IP **令牌桶**：容量 `C`、填充 `r`/秒 → 最大稳态 `r`、最大突发 `C`，内存 `O(不同 IP 数)` 并加 LRU 上限（如 10k IP），使限流器自身不成为内存 DoS 向量。默认建议：`/model-gateway/test` 10/min；`/workflow/runs` 30/min；`/memory/v2/search`、`/memory/v2/capsules` 60/min。

**明确标注：单进程内存限流。多进程共享限流留到 v1.0。**

## 6. 可行性判定

每个 v0.9.6 任务都是 **加法式或受测试保护** 的；两个有风险的性能任务（T3/T4）位于 T1 基线 + T2 测试网之后；头条收益（N+1）用计数即可证明。没有任务需要 async、Repository 或端点拆分——**与推迟的高 ρ 工作零重叠**。

后续路线：**v0.9.7** = async/aiosqlite + cache（在已测、已测量的基座上）；**v1.0** = 拆 61 端点 God Object + Repository/DI（在完整覆盖上）。
