## 🎯 设计决策与实施细节

### 决策 1：渐进式迁移策略

**问题**：如何从内存字典平滑迁移到数据库持久化？

**方案**：
```python
# 双写策略
def create_run(req):
    run = build_run(req)
    
    # 1. 写入数据库（新）
    persistence.save_run(run_id, run)
    
    # 2. 写入内存（旧，保留为 fallback）
    _RUNS[run_id] = run
    
    return run

# 读优先策略
def get_run(run_id):
    # 1. 优先从数据库读取
    run = persistence.get_run(run_id)
    if run:
        return run
    
    # 2. Fallback 到内存
    return _RUNS.get(run_id) or {"error": "not_found"}
```

**优势**：
- 零停机迁移
- 向后兼容
- 逐步验证后可移除内存字典（v0.9.6）

### 决策 2：数据库表设计

**设计原则**：
- 完整 JSON 存储（`run_data TEXT`）+ 索引字段提取
- 避免复杂的关系表（简化查询）
- 支持高效的时间范围查询（TTL 清理）

**表结构**：
```sql
CREATE TABLE workflow_runs (
    run_id TEXT PRIMARY KEY,           -- 主键
    scenario TEXT NOT NULL,            -- 索引字段
    status TEXT NOT NULL,              -- 索引字段
    created_at TEXT NOT NULL,          -- 索引字段（TTL）
    run_data TEXT NOT NULL,            -- 完整 JSON
    -- 其他提取字段用于快速过滤
);

CREATE INDEX idx_workflow_runs_created_at ON workflow_runs(created_at);
CREATE INDEX idx_workflow_runs_status ON workflow_runs(status);
CREATE INDEX idx_workflow_runs_scenario ON workflow_runs(scenario);
```

**权衡**：
- ✅ 简单：单表，易于理解和维护
- ✅ 灵活：完整 JSON 保留所有信息
- ✅ 高效：索引支持常见查询模式
- ⚠️ 冗余：部分字段重复（可接受的权衡）

### 决策 3：Datetime 工具设计

**提供 4 种格式**：

1. `utc_now()` - 标准 timezone-aware datetime（推荐）
2. `utc_now_iso()` - 带时区的完整 ISO 格式
3. `utc_now_iso_compact()` - 紧凑格式（文件名友好）
4. `utc_timestamp_legacy()` - 向后兼容格式

**理由**：
- 不同场景需要不同格式
- 明确命名避免混淆
- Legacy 格式支持平滑迁移

### 决策 4：Lifespan 实现

**采用 async contextmanager 模式**：
```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup
    try:
        init_db()
        init_workflow_persistence()
    except Exception:
        pass  # 容错处理
    
    yield
    
    # Shutdown（预留）
    pass
```

**优势**：
- 符合 FastAPI 最新最佳实践
- 为将来的 shutdown 逻辑预留空间
- 消除 deprecation warnings

## 📈 性能影响分析

### 持久化性能

**写入性能**：
- 内存字典：~1μs
- SQLite 插入：~50-100μs
- **影响**：可忽略（workflow 创建本身耗时 200ms+）

**读取性能**：
- 内存字典：~1μs
- SQLite 主键查询：~10-20μs
- **影响**：可忽略（网络延迟 >>20μs）

**存储占用**：
- 单个 run：~5-10KB
- 1000 个 runs：~5-10MB
- **结论**：7 天 TTL 足够，磁盘占用可控

### 并发能力

**多进程支持**：
- v0.9.4：❌ 内存字典不共享
- v0.9.5：✅ SQLite 支持多进程读写

**锁竞争**：
- SQLite 默认使用文件锁
- 读操作并发无锁
- 写操作短暂持锁（~100μs）
- **结论**：适合中等并发场景（<100 QPS）
