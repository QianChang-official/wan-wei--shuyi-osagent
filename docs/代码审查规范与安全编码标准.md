# 宛委·万枢 代码审查规范与安全编码标准

> 版本：v1.0（第四批修复后建立）
> 适用范围：`wan-wei--shuyi-osagent` 后端（FastAPI）/ 前端（Vue3+TS）/ 桌面端（Electron）
> 目标：把本轮全量审计发现的问题（审计报告 00–13 章口径，见 audit-report）的共性根因固化为团队红线，
> 确保后续开发不再引入同类问题。

---

## 一、审查流程（Code Review）

### 1.1 审查清单（每次 PR 必过）

| 类别 | 检查项 | 红线 |
|---|---|---|
| **鉴权** | 新增端点是否默认鉴权？是否在 `_PUBLIC_PATHS` 白名单中显式豁免？ | 禁止新增未鉴权端点 |
| **敏感数据** | 密钥/令牌/密码是否加密落盘？响应是否脱敏？日志/审计是否打码？ | 禁止明文落盘、明文回显 |
| **输入校验** | 请求模型是否 `extra='forbid'`？枚举值是否服务端校验？ | 禁止信任客户端自报值 |
| **事务完整性** | 写路径是否 `commit/rollback` 成对？是否有悬挂事务风险？ | 禁止裸写无回滚 |
| **审计留痕** | 关键操作（创建/删除/越权/策略拦截）是否落审计？ | 禁止静默失败 |
| **诚实标注** | stub/mock/simulated 数据是否在响应/UI 中明确标注？ | 禁止伪装真实数据 |
| **测试** | 新增功能是否带回归测试？安全修复是否带安全用例？ | 禁止无测试合并 |

### 1.2 审查角色分工

- **作者自查**：提交前跑全量测试 + 本规范 §2 安全红线逐条核对。
- **同行审查**：至少 1 名同级工程师，重点看契约一致性与边界条件。
- **安全审查**：涉及鉴权、加密、策略、外部调用的 PR 必须过安全清单（§2）。

## 二、安全编码红线（Security Baseline）

以下规则源于本次审计的真实漏洞，每条都附「反例 → 正例」。

### 2.1 鉴权与门禁

**R-01：默认保护，显式公开。** 新端点默认要求 API key；公开路径必须加入
`_PUBLIC_PATHS` 白名单并注明理由。禁止黑名单反向枚举。

**R-02：档位/权限以服务端状态为准。** 请求体携带的 `gear`、`role`、`level`
等权限字段仅作参考，服务端必须重新解析并校验。
反例：`agents.py` 直接信任 `body.gear='device'`（P1）。
正例：`require_gear()` 统一门禁 + 未授权降级 + 审计（`guards.py`）。

**R-03：危险档默认禁用。** device 档、真实写操作、外部调用等高危能力，
默认关闭，需显式环境变量/配置开启，且开启后全程审计。

### 2.2 敏感数据生命周期

**R-04：密钥加密落盘。** 任何 API key、token、password 落盘前必须走
`encryption.encrypt`，读取侧透明解密；密文带版本前缀（`enc:v1:`）便于轮换。
存量明文采用惰性迁移：首次读取时自动加密回写。

**R-05：响应侧永不回显敏感值。** 列表/详情接口对 `env`、`api_key` 等字段
只回键名或尾 4 位；调用计划/审计记录对敏感键打码（`******`）。

**R-06：日志与审计截断。** 入库的 `arguments`、`payload` 做大小截断
（≤400 字）+ 敏感键打码，防止审计表膨胀与密钥泄露。

### 2.3 策略与内容安全

**R-07：用户输入过 Policy Gate。** 任何持久化用户文本（记忆、指令、常用语、
知识文档）写入前必须 `evaluate_policy`；reject/quarantine → 422 并落审计。

**R-08：自然语言绕过防御。** 敏感词检测须覆盖中文同义表述（口令/密钥/凭据 +
赋值动词）、间隔符拆写（`p a s s w o r d`）、常见注入变体
（`忽略之前的指令` / `disregard guidelines`）。新增绕过样本须补进回归集。

### 2.4 数据完整性

**R-09：写路径事务闭环。** 所有 DB 写操作必须 `try/commit/except/rollback`；
多步写用 `transaction()` 上下文管理器。
反例：`main.py:506-508` 4 条 INSERT 仅末尾 commit（P1 脏事务泄漏）。

**R-10：热路径禁跑 DDL。** schema 迁移/建表只允许模块级 once 或 lifespan
一次性执行，禁止在请求热路径重复跑（`capsule_store.py` 教训）。

### 2.5 外部调用与子进程

**R-11：子进程最小环境。** `subprocess.Popen` 的 `env` 只传必要变量，
禁传完整 `os.environ`（防密钥泄漏到子进程）。

**R-12：路径与命令白名单。** 文件路径、shell 命令、git 操作必须有白名单
校验；用户可控路径做 realpath 越界检测。

### 2.6 前端安全

**R-13：API Key 不硬编码。** 前端不存密钥到 localStorage 明文；注入链路
统一走 `setPlatformApiKey`；生产包不含 dev key。

**R-14：降级如实标注。** 后端不可用时前端显示离线/模拟标识，禁止用
假数据冒充真实响应。

## 三、测试基线

### 3.1 覆盖率要求

- **安全修复**：必须附回归测试（正面 + 反面 + 边界）。
- **新端点**：至少 1 条 TestClient 级冒烟（CRUD happy path + 1 个边界断言）。
- **策略/门禁变更**：必须补绕过用例到 `security_cases.jsonl` 或对应测试文件。

### 3.2 测试隔离

- 每个测试用 `WANWEI_PLATFORM_DIR` + `WANWEI_MEMORY_DB` 指向 `tmp_path`，
  禁止污染本机 `data/`。
- 模块级 `JsonStore`/`db.get_conn` 等单例，测试用 `monkeypatch` 重定向，
  禁止依赖测试执行顺序。

### 3.3 CI 接入

- 后端：`python -m pytest --basetemp .pytest-ci -p no:cacheprovider` 全绿；当前基线以 `docs/PROJECT_AUDIT_MANIFEST.md` 与最近一次 CI 为准。
- 前端：`npm run build` 0 错误，`npm run test:security` 与 `npm run test:contracts` 通过。
- 桌面端：`npm test` 通过。
- 安全用例集（`data/eval/security_cases.jsonl`）每次安全相关 PR 必跑。

## 四、架构约定

- **platform_api 模块**：统一走 `guards.py` 的 `audit_safe()` 落审计，
  `require_gear()` 做档位门禁；不直接调 `audit.service.record`。
- **加密**：统一 `app.security.encryption`，密钥派生优先级
  `WANWEI_ENCRYPTION_KEY` > API key 派生。
- **脱敏**：统一 `app.security.redaction` + `guards.mask_secret_keys`。
- **存储**：`JsonStore` 路径惰性解析（随环境变量漂移），测试可安全重定向。

---

## 五、变更流程

1. 安全相关 PR 标题前缀 `[security]`，并在描述中注明触发的红线编号。
2. 新增红线须经团队评审后补入本文档，并同步补回归测试。
3. 每季度复审本文档，结合实际绕过样本更新 §2.3 策略规则。
